
## **1. What are the different ways to manage state in React?**

React offers multiple methods to manage state depending on complexity:

### **A. Local State (Component-level)**

- Uses `useState`, `useReducer`
    
- Suitable for small, isolated state
    

### **B. Lifted State (Sent to parent)**

- Moving state up the component tree so multiple children can share it
    

### **C. Context API (Global state without prop drilling)**

- Good for theme, user auth, language, or small global needs
    

### **D. Redux (External state management library)**

- Suitable for large/complex apps
    
- Predictable global state container
    

### **E. Zustand, Jotai, Recoil (Modern alternatives)**

- Lightweight, simpler than Redux in many cases
    

### **F. Server State (React Query, SWR)**

- Manages data fetched from APIs
    
- Handles caching, refetching, sync with server

## **2. What is Context API?**

Context API provides a way to share data **globally** between components **without passing props manually at every level (no prop drilling)**.

### Example:

```jsx
const UserContext = createContext();

function App() {
  return (
    <UserContext.Provider value={{ name: "Shubham" }}>
      <Profile />
    </UserContext.Provider>
  );
}

function Profile() {
  const user = useContext(UserContext);
  return <h1>{user.name}</h1>;
}
```

### Use cases:

- Auth data
    
- Theme (light/dark)
    
- Localization/language
    
- User preferences

## **3. When should you use Redux instead of Context API?**

Use **Redux** when the application becomes complex and needs:

### **Use Redux when:**

- Many components need the same state
    
- State transitions are complex
    
- You want centralized logic for updating state
    
- You need powerful dev tools like **Redux DevTools**
    
- You need middleware (e.g., Redux Thunk, Saga)
    
- You want predictable and testable state management
    

### **Do NOT use Redux when:**

- State is small/simple
    
- Prop drilling occurs only at 1–2 levels
    
- Context API is enough for your use case
    

**Rule of thumb:**  
Small/medium apps → **Context**  
Large/enterprise apps → **Redux**

## **4. What are reducers in Redux?**

Reducers are **pure functions** that take:

- Current **state**
    
- An **action**
    

and return a **new state**.

### Example:

```jsx
function counterReducer(state = { count: 0 }, action) {
  switch (action.type) {
    case "INCREMENT":
      return { count: state.count + 1 };
    case "DECREMENT":
      return { count: state.count - 1 };
    default:
      return state;
  }
}
```

### Important characteristics:

- Pure (no side effects)
    
- Returns new state (does not mutate old state)
    
- Based on action type

## **5. What is the role of actions and dispatch in Redux?**

### **Actions**

Actions are **objects** describing _what happened_.  
They **must have a type**, and can have additional data (payload).

### Example action:

```js
{ type: "ADD_TODO", payload: "Learn Redux" }
```

---

### **Dispatch**

`dispatch()` sends an action to the Redux store.

### Example:

```js
dispatch({ type: "INCREMENT" });
```

### Flow:

1. Component dispatches an action
    
2. Reducer receives action
    
3. Reducer updates and returns new state
    
4. UI re-renders based on updated state
    

This flow creates a **predictable, centralized state management system**.
